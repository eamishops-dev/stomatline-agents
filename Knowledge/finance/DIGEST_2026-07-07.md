# Finance — 2026-07-07
### FIN-1 (v2). Маржинальний шлюз: накладна нижче порогу не підтверджується без керівника
Доопрацювання за Request changes Аміра (2026-07-06): «я б краще блокував підтвердження накладної при певній маржінальності і давав цю накладну на погодження керівнику».

Користувач: Амір (керівник) + менеджер продажу.
Біль (час/енергія/гроші): v1 показувала колір, але рішення по червоному рядку лишалося за менеджером — м'який контроль: мінусовий продаж усе одно міг піти в роботу, Амір дізнавався постфактум зі звіту (гроші); вибіркова ручна перевірка документів керівником (час/енергія).

Рішення (що змінилося проти v1):
1. Світлофор лишається — ранній сигнал у момент набору рядків SO.
2. Замість «менеджер сам підтверджує червоне» — жорсткий шлюз: документ із валовою маржею нижче порогу НЕ підтверджується. Кнопка «Підтвердити» заблокована; єдина доступна дія — «Надіслати на погодження керівнику».
3. Керівник отримує в Odoo activity з контекстом: клієнт, рядки нижче порогу, собівартість, маржа %, історія клієнта. Одна кнопка: Погодити (разовий виняток) або Повернути (менеджер коригує ціну).
4. Пороги, два рівні: жовтий = нижче цільової маржі категорії (видно, не блокує); червоний = нижче мінімального порогу (блокує). Стартовий поріг Амір задає одним числом; пороги по категоріях — з фактичних даних першого місяця.
5. Журнал винятків: кожне погодження фіксується (хто, для кого, скільки маржі віддали, причина). Раз на місяць — список клієнтів/SKU, що системно тиснуть на поріг → перегляд ціни або собівартості.

Доставка в робочий процес: точка шлюзу №1 — підтвердження SO (ловить до резервування складу); точка №2, страхувальна — validate видаткової накладної/рахунку, якщо ціни змінилися після підтвердження. Реалізація: Odoo Sales Margin (з v1) + одне approval-правило на кнопку підтвердження (Studio rule або server action) + activity керівнику. Нових модулів нуль.
Захист потоку: блок не зупиняє відвантаження вже підтвердженого — шлюз стоїть на підтвердженні; SLA погодження — того ж дня, з телефону одним тапом; заблоковані документи видно списком «Очікують погодження» на дашборді керівника.

Режим виконання: Гібрид (система блокує і готує контекст; виняток погоджує людина-керівник).
Цифра ефекту: даних ще нема — чесно. Механіка ефекту: все, що раніше проходило з маржею нижче порогу, тепер або репрайситься (маржа повертається), або стає свідомим задокументованим винятком. Побічний актив: журнал винятків = доказ цінової дисципліни для due diligence. Чого бракує в Odoo: landed cost на приходах (з v1) + поле «мінімальний поріг маржі» (одне число, далі по категоріях).
Метрика болю: заблокованих документів/тиждень; % повернутих на репрайс vs погоджених винятків; грн маржі, повернутої репрайсом; медіанний час погодження (ціль < 2 год — інакше шлюз шкодить сервісу і довірі).

## Зв'язка дня: ODOO-4 (дебіторка) вже закладає той самий патерн «блок + погодження керівника» через credit limit на SO — шлюз маржі і шлюз кредиту мають жити на одній кнопці підтвердження, щоб менеджер бачив одне вікно погодження, а не два.
## Якісний шлюз: 78→96/100. Скептик: «жорсткий блок уповільнить відвантаження і вдарить по довірі» → шлюз на підтвердженні (не на відвантаженні), SLA того ж дня, мобільне погодження одним тапом. Спрощувач: нуль нових модулів — Sales Margin + одне approval-правило; пороги стартують з одного числа. Користувач: менеджер одразу бачить, до якої ціни підняти, щоб пройти без погодження; керівник вирішує одним тапом.
