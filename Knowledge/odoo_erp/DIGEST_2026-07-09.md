# Odoo ERP — 2026-07-09

### ODOO-6. Платіж знаходить свій рахунок: виписка розноситься сама, дебіторка не бреше
Користувач: Фінансист (щоденне рознесення платежів); Фермер (не дзвонить клініці за вже оплачений рахунок); опосередковано — клініка, яка ніколи не отримує нагадування за те, що вже сплатила.
Біль (час/енергія/гроші): гроші від клінік заходять на рахунок щодня, але в Odoo вони зʼявляються лише після ручного рознесення: відкрити клієнт-банк, знайти платіж, вгадати рахунок (призначення платежу — «оплата за матеріали», без номера), зареєструвати payment, звірити. Час: 30–60 хв/день монотонного клікання, при 1000 клінік ×2–4 платежі/міс — це 100+ платежів/день, ручне рознесення фізично неможливе. Енергія: робота-зомбі з ризиком помилки — платіж клініки з кількома відкритими рахунками легко ліг не туди. Гроші — найдорожче: поки платіж не рознесено, дебіторка ЗАВИЩЕНА і бреше всім, хто на неї дивиться: драбина follow-up ODOO-4 надсилає нагадування за вже оплачений рахунок (прямий удар по довірі — Strategy: довіра перед прибутком), каса FIN-2 будує 13 тижнів на застарілому факті, Фермер починає незручну розмову, яка не мала статися. Функція «облік надходжень» сидить у режимі «Людина» і є вузьким місцем правдивості всього order-to-cash.
Рішення: платіж матчиться з рахунком сам. (1) Щоденний ранковий імпорт виписки з клієнт-банку в Odoo — CSV/XML-експорт є в кожному укрбанку (Приват24 для бізнесу, monobank, Ощад), імпорт виписок в Odoo вбудований; жодного банківського API на старті. (2) Reconciliation models (вбудовано): каскад правил — точний матч за номером рахунку в призначенні платежу → матч за ЄДРПОУ/контрагентом + сумою → часткова оплата як partial reconcile; система пропонує, впевнені матчі проводить. (3) Безкоштовний підсилювач точності: у шаблон рахунку додається рядок «У призначенні платежу вкажіть: INV-XXXX» — % автоматчу росте з кожним місяцем звички. (4) Automated action: платіж без матчу >24 год → activity Фінансисту з контекстом; переплата/аванс → пропозиція зарахувати на аванси клініки (підтверджує людина). (5) Наслідок без жодного налаштування: рознесений рахунок автоматично випадає з драбини нагадувань ODOO-4 — хибне нагадування стає неможливим за побудовою.
Доставка в робочий процес: модуль Accounting — bank statement import + reconciliation models (усе вбудовано) + 1 automated action (сторожа нематчів) + рядок у шаблоні рахунку. Додати: дашборд «Каса дня» (надійшло / рознесено автоматично / винятки) — ранковий екран Фінансиста поруч з Aged Receivable. Кастомних модулів нуль.
Режим виконання: був — Людина (щоденне ручне рознесення, лаг у дні, помилки). Стане — Гібрид (система матчить ≥85% сама, людина розбирає короткий список винятків); шлях до AI — автозарахування типових винятків після 3 місяців статистики причин.
Метрика болю: % платежів, рознесених автоматично (ціль ≥85%), час на виписку/день (30–60 хв → ≤10 хв), лаг «гроші в банку → видно в Odoo» (дні → ранок наступного дня), нагадувань за вже оплачені рахунки/міс (→ 0 за побудовою).

## Зв'язка дня: ODOO-4 (дебіторка на автопілоті) — драбина follow-up чесна лише тоді, коли оплати розносяться того ж ранку; я закриваю дірку, яку ODOO-4 сам позначив «окремою майбутньою фігурою». Бонус: FIN-2 (каса на 13 тижнів) отримує щоденний ФАКТ надходжень замість тижневого зведення — прогноз коригується щоранку; SALES-1 — Фермер у ранковому брифі бачить «клініка сплатила вчора», привід для подяки замість нагадування.

## Якісний шлюз: 75→96/100. Лінзи:
Скептик: «платежів поки небагато — 20 хвилин руками, біль дрібний» → біль не в хвилинах, а в лазі: кожен день між грошима в банку і записом в Odoo — це день, коли дебіторка бреше, і одне хибне нагадування клініці коштує більше довіри, ніж рік зекономлених хвилин; при цільових 1000 клінік ручне рознесення неможливе фізично — звичка і каскад правил будуються зараз, коли винятки можна розібрати поштучно. «Банківські API в Україні — складно і довго» → API не потрібен: щоденний файловий експорт із клієнт-банку + вбудований імпорт; API — окреме майбутнє покращення, не передумова. Спрощувач: нуль інтеграцій, нуль кастому — імпорт + reconciliation models + 1 automated action + рядок у шаблоні рахунку; прибрано: онлайн-синхронізацію з банком (передчасно), автозарахування авансів без людини (ціна помилки в чужих грошах вища за хвилину підтвердження), автопривʼязку «одним платежем за кілька рахунків» FIFO (лишається людині — див. питання вечора). Користувач: Фінансист о 9:05 бачить «34 платежі: 31 рознесено, 3 винятки» і за 10 хвилин закриває день замість години клікання; Фермер відкриває бриф і бачить свіжий платіжний статус клініки, а не вчорашню правду.

## Питання вечора: клініка платить одним платежем за кілька рахунків без призначення — розносити автоматично від найстарішого (FIFO, швидкість) чи завжди лишати людині (точність і сигнал, що клініці незручно платити по-рахунково)?
