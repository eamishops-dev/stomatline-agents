# Odoo ERP — 2026-07-06

### ODOO-1. Паспорт SKU: жодного товару без повної картки
Користувач: Закупщик і Оператор складу (внутрішні); опосередковано — клініка, що отримує саме те, що замовила.
Біль (час/енергія/гроші): картки товарів заповнюються «як вийшло»: дублікати SKU, немає виробника, одиниць виміру, партій і термінів придатності. Закупщик щоразу вручну звіряє накладну постачальника з каталогом (час), Оператор вгадує, який із двох однакових композитів брати (енергія, помилки комплектації), компанія списує прострочене, бо FEFO неможливий без дат (гроші). Головне: на брудному довіднику жодна майбутня AI-функція (прогноз дозамовлення, виявлення аномалій) не запуститься — все застрягає в режимі «Людина».
Рішення: стандарт «Паспорт SKU» — 8 обов'язкових полів картки: internal reference (єдиний формат), виробник, категорія, од. виміру закупівлі/зберігання, tracking=Lots + Expiration Dates, постачальник(и) з цінами, min/max правило, фото. Server action в Odoo блокує активацію товару з неповним паспортом; nightly automated action шукає дублікати за назвою/штрихкодом.
Доставка в робочий процес: модуль Inventory → Traceability: увімкнути Lots & Expiration Dates + Removal Strategy = FEFO; Product Template → required fields через server action (без кастомних модулів); дашборд «Якість довідника» (kanban/список: товари з неповним паспортом, знайдені дублікати) — перший екран Закупщика.
Режим виконання: був — Людина (кожен заповнює як хоче, ніхто не перевіряє). Стане — Гібрид (людина заповнює, система блокує неповне і сама знаходить дублікати).
Метрика болю: % активних SKU з повним паспортом (ціль 100%), кількість дублікатів у довіднику (ціль 0), грн списань за простроченням/міс.
