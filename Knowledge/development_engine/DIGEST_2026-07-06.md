### DEV-1. Правило вхідних воріт: сутність не входить у систему неповною
Користувач: кожен, хто створює записи в Odoo; Амір як власник якості даних.
Біль (час/енергія/гроші): патерн з _index: неповні дані на вході вилазять у різних агентів під різними іменами — товар без паспорта (ODOO-1), прихід без партій (WH-1), собівартість без доставки (FIN-1). Кожен пропуск апстрім = години ремонту даунстрім.
Рішення: одна системна зміна: для 4 сутностей (товар, клініка, замовлення, прихід) визначено «мінімальний паспорт» + server action, що не пускає неповний запис в активний стан. Джерело правди по полях — одна сторінка «Паспорти даних» в Odoo Knowledge.
Доставка в робочий процес: server actions по 4 сутностях; draft зберігати можна, активувати неповне — ні; онбординг нової людини починається зі сторінки паспортів.
Режим виконання функції: був Людина (дисципліна на совісті) → стане Гібрид (система не пропускає).
SOP-блок: 1) створюєш запис → заповни поля паспорта; 2) не знаєш значення — постав «?» і задачу власнику довідника; 3) система не активує неповний запис; 4) активація = запис готовий для всіх наступних функцій.
Метрика болю: % активних записів з повним паспортом по 4 сутностях; час новачка до першого самостійного запису.
Патерн-джерело: ODOO-1 (approved), WH-1, FIN-1 — сьогоднішні.
